# مستند تحليل النظام (System Analysis) - مشروع تحويل صور الكتب العربية إلى نصوص مقروءة (Arabic OCR System)

## 1. نظرة عامة (System Overview)
يهدف هذا النظام إلى توفير منصة متكاملة وفعالة لتحويل صور الكتب والمستندات المكتوبة باللغة العربية إلى نصوص رقمية يمكن قراءتها، البحث فيها، وحفظها. 
يبدأ سير العمل (Workflow) بالخطوات التالية:
1. **رفع الكتاب أو الصور**: يقوم المستخدم من خلال واجهة المستخدم (React) برفع ملف للكتاب كاملاً (مثل PDF) أو مجموعة صور تمثل صفحات الكتاب.
2. **تقسيم الكتاب وإنشاء السجلات**: يستقبل الخادم (FastAPI) الطلب. إذا كان المرفوع كتاباً كاملاً، يقوم النظام بتقسيمه واستخراج صفحاته كصور مستقلة. ثم يُنشئ سجلاً للكتاب وسجلاً لكل صفحة (صورة) في قاعدة البيانات المرتبطة بالكتاب.
3. **بدء المعالجة**: يقوم المستخدم بطلب بدء عملية استخراج النص (OCR).
4. **المعالجة في الخلفية**: يتم إرسال المهام إلى طابور المعالجة (Task Queue) لتنفذ بشكل غير متزامن (Asynchronous)، حيث تمر الصور عبر عمليات التحسين والمعالجة المسبقة ثم تُمرر لنموذج الذكاء الاصطناعي (PaddleOCR).
5. **متابعة الحالة**: يمكن للمستخدم متابعة حالة المعالجة (قيد الانتظار، جاري المعالجة، مكتمل، فشل) عبر الواجهة.
6. **عرض النتائج والتحرير**: بمجرد اكتمال العملية، يعرض النظام النص المستخرج بجانب صورة الصفحة الأصلية لتسهيل المراجعة والتعديل.
7. **الطباعة والتصدير**: يتيح النظام إمكانية طباعة صفحة محددة مع نصها، أو طباعة/تصدير كافة نصوص صفحات الكتاب كملف واحد.

## 2. بنية النظام (System Architecture)
يعتمد النظام على بنية خادم-عميل (Client-Server Architecture) مع فصل واضح للمهام:

* **الواجهة الأمامية (Frontend - React)**: تتولى تفاعل المستخدم، عرض واجهات الرفع، لوحة التحكم لمتابعة حالة معالجة الكتب، وعرض النتائج النهائية للنصوص. تتواصل مع الخادم عبر طلبات HTTP/RESTful APIs.
* **الواجهة الخلفية (Backend - FastAPI)**: تمثل واجهة برمجة التطبيقات الأساسية وموجه الطلبات. تتلقى الملفات، تدير قاعدة البيانات، وتوجه مهام الذكاء الاصطناعي للاشتغال في الخلفية. يتميز FastAPI بالسرعة والدعم المدمج للعمليات غير المتزامنة (Async).
* **طبقة الخدمات (Services Layer & AI Model)**: تحتوي وبشكل معزول على محرك الذكاء الاصطناعي `PaddlePaddle/arabic_PP-OCRv3_mobile_rec`. يتم استدعاء هذه الطبقة من خلال "مُشغّل مهام خلفية" (مثل Celery أو Redis Queue، أو BackgroundTasks المدمجة في FastAPI كبداية) لضمان عدم توقف الـ API (Blocking) أثناء فترة تطبيق المعالجة العميقة للصور.

## 3. مخطط قاعدة البيانات (Database Schema)
سيتم استخدام قاعدة بيانات علائقية (PostgreSQL للمنظومة الإنتاجية أو SQLite لبيئة التطوير). التصميم يحتوي على كيانات أساسية:

### جدول الكتب (`Books`)
يمثل الكتاب أو الملف الرئيسي المرفوع.
* `id` (Primary Key, UUID/Int): المعرف الفريد للكتاب.
* `title` (String): عنوان الكتاب (يحدده المستخدم أو افتراضي).
* `created_at` (DateTime): تاريخ الرفع.
* `status` (Enum/String): حالة معالجة الكتاب الكلية (Pending, Processing, Completed, Failed).

### جدول الصفحات (`Pages`)
يمثل الصور المرفوعة التابعة لكتاب معين.
* `id` (Primary Key, UUID/Int): المعرف الفريد للصفحة.
* `book_id` (Foreign Key -> Books.id): الكتاب المرتبط.
* `page_number` (Int): ترتيب الصفحة داخل الكتاب.
* `image_path` (String): مسار تخزين صورة الصفحة على الخادم أو التخزين السحابي.
* `status` (Enum/String): حالة معالجة الصفحة (Pending, Processing, Completed, Failed).

### جدول نتائج الاستخراج (`OCR_Results`)
يحتفظ بالنصوص المستخرجة لكل صفحة مع تفاصيل إضافية.
* `id` (Primary Key, UUID/Int): المعرف الفريد للنتيجة.
* `page_id` (Foreign Key -> Pages.id): الصفحة المرتبطة.
* `extracted_text` (Text): النص العربي الذي تم استخراجه بواسطة نموذج OCR.
* `confidence_score` (Float): نسبة دقة التعرف (Confidence) المسترجعة من النموذج.
* `created_at` (DateTime): توقيت اكتمال استخراج النص.

*ملاحظة: العلاقة بين `Books` و `Pages` هي (1-to-Many)، وبين `Pages` و `OCR_Results` هي (1-to-1) لحفظ نص كل صفحة لتسهيل الاسترجاع والتعديل.*

## 4. تصميم واجهات برمجة التطبيقات (API Design)
تستخدم واجهات RESTful عبر إطار `FastAPI`:

1. **رفع كتاب (PDF) أو صور صفحات جديدة**
   * **المسار**: `POST /api/v1/books/upload`
   * **المدخلات**: `Multipart Data` (ملف الكتاب PDF أو ملفات الصور المتعددة، تفاصيل مساعدة مثل عنوان الكتاب).
   * **المخرجات**: `book_id` ورسالة تأكيد استلام الملف/الصور والبدء بتقسيمها (في حال PDF) بنجاح.

2. **بدء عملية استخراج النص (OCR Trigger)**
   * **المسار**: `POST /api/v1/books/{book_id}/process`
   * **المدخلات**: `book_id` (عبر المسار).
   * **المخرجات**: إرجاع رسالة تأكيد بأن العملية أُحيلت للمعالجة في الخلفية.

3. **الاستعلام عن حالة المعالجة (Status Check)**
   * **المسار**: `GET /api/v1/books/{book_id}/status`
   * **المخرجات**: تفصيل حالة الكتاب (`status`) وحالة معالجة كل صفحة متعلقة به ونسبة الإنجاز التقديرية (مثلاً: تمت معالجة 5 من 10 صفحات).

4. **جلب النصوص المستخرجة للكتاب**
   * **المسار**: `GET /api/v1/books/{book_id}/results`
   * **المخرجات**: إرجاع قائمة مرتبة برقم الصفحة، تحتوي على `extracted_text` بالإضافة إلى مسار الصورة `image_path` لعرضها جنباً إلى جنب.

5. **تصدير أو طباعة النصوص (Export/Print)**
   * **المسار**: `GET /api/v1/books/{book_id}/export`
   * **المدخلات**: `format` (مثل `pdf`, `txt`, `docx`) بالإضافة إلى `page_id` اختياري لطباعة صفحة معينة.
   * **المخرجات**: ملف جاهز للتحميل والطباعة يحتوي على النصوص المستخرجة بشكل منسق.

## 5. خطوات المعالجة والذكاء الاصطناعي (AI & OCR Pipeline)
يمر خط المعالجة بالمراحل المتسلسلة التالية:
1. **المعالجة الأولية وتقسيم الكتاب**: إذا كان المرفوع مستند PDF كامل، يتم استخدام مكتبة (مثل `pdf2image` أو `PyMuPDF`) لتقسيم الكتاب وتحويل كل صفحة إلى صورة مستقلة. تُحفظ هذه الصور بالخادم وتسجل مساراتها في جدول `Pages`.
2. **استقبال المهمة**: عند بدء عملية (OCR Trigger)، يقوم عامل الخلفية (Background Worker) بسحب مهمة معالجة `book_id`.
3. **جلب الصور**: قراءة مسار `image_path` لكل صفحة متعلقة بالكتاب من قاعدة البيانات.
4. **المعالجة المسبقة للصور (Pre-processing)**: 
   * تحويل الصورة للتدرج الرمادي (Grayscale).
   * إزالة الضوضاء (Noise Reduction) وتطبيق التحول الثنائي (Binarization).
   * التعديل التلقائي لميلان الصفحة (Deskewing) لزيادة دقة التعرف.
5. **تطبيق نموذج PaddleOCR**: 
   * تمرير الصورة المعالجة إلى محرك ذكاء اصطناعي مُحمل بنموذج `arabic_PP-OCRv3_mobile_rec`.
   * يتولى المحرك اكتشاف أماكن النصوص (Text Detection) أولاً، ثم التعرف عليها (Text Recognition).
6. **المعالجة اللاحقة (Post-processing)**: تجميع النصوص الناتجة، تنظيف الفراغات، وتصحيح التشفير إن لزم؛ للتوصل إلى نص عربي مفهوم متصل.
7. **الحفظ**: تخزين النص النهائي ومعامل الثقة (Confidence) في جدول `OCR_Results` وتحديث حالة `Pages` و `Books` إلى `Completed`.

## 6. التحديات التقنية والحلول المقترحة

* **التعامل مع الصور رديئة الجودة**:
   * *المشكلة*: صور الهواتف قد تكون مهتزة، بإضاءة خافتة، أو تحتوي على ظل ودرجات تباين سيئة.
   * *الحل المقترح*: استخدام مكتبة `OpenCV` في مرحلة التجهيز المسبق (Pre-processing Pipeline) لبناء خوارزميات تعديل التباين التكيفي (Adaptive Thresholding/CLAHE)، وتصحيح المنظور هندسياً (Perspective Correction) لقص حواف الصفحة بشكل مثالي قبل تمريرها لشبكة استخراج النصوص.

* **التعامل مع العمليات التي تستغرق وقتاً طويلاً (Asynchronous Task Processing)**:
   * *المشكلة*: تطبيق الـ OCR على كتاب كامل (مئات الصفحات) يأخذ وقتاً حاسوبياً طويلاً، مما قد يسبب انقطاع الـ Request Timeout بالواجهة وتجميد السيرفر إن تمت المعالجة فورياً (Synchronously).
   * *الحل المقترح*: استخدام هندسة الطوابير (Message Brokers) وتطبيق وسيط رسائل مثل `Redis` أو `RabbitMQ` متصلاً بعمال `Celery`. يقوم الـ API الرئيسي بإعادة استجابة سريعة فور استلام الطلب ويُرسل المهام إلى الخلفية. تعتمد الواجهة الأمامية على تقنية السحب المستمر (Polling) للاستعلام عن الحالة كل بضع ثوانٍ (عن طريق مسار Status Check)، أو يمكن تطبيق `WebSockets` لتلقي إشعارات فورية متزامنة من السيرفر بمجرد انتهاء معالجة الصفحة وإنجاز المهمة كلياً.
